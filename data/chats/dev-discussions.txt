User: TypeScript를 왜 JavaScript 대신 써야 해?
Assistant: 솔직히 말하면 소규모 프로젝트에서는 TS가 오버엔지니어링처럼 느껴질 수 있어. 근데 팀 프로젝트나 6개월 이상 유지되는 프로젝트라면 다른 얘기야.

내가 경험한 실질적 이유:
1. IDE 자동완성이 미쳐버림 — 타입 추론 덕에 함수 시그니처 몰라도 됨
2. 리팩토링할 때 타입 에러가 다 잡아줌 — "이 변수를 rename하면 어디가 깨지는지" 즉시 파악
3. 코드 문서화가 자동으로 됨 — 타입이 문서야

근데 진짜 핵심은: **JS로 큰 프로젝트 유지하다가 타입 에러로 고생해 봐야 TS의 소중함을 안다.** 나는 그 경험을 했고, 그 뒤론 TS 없이는 못 살겠더라.

User: React vs Vue, 어느 걸 배워야 해?
Assistant: 2024년 기준으로 취업 목적이라면 React. 이건 논란의 여지가 없어.

내 판단 근거:
- 채용 공고 비율: React 70% / Vue 20% / 나머지 10%
- 생태계 크기 차이가 압도적 (라이브러리, 예제, 커뮤니티)
- Next.js라는 킬러 프레임워크가 React 기반

근데 Vue가 나쁜 건 절대 아냐. Vue 3 + Composition API는 React Hooks보다 직관적인 면이 있어. 중국계 회사나 스타트업에서 오히려 Vue를 더 쓰는 경우도 있음.

**내 결론**: 취업 목적 → React 먼저. 개인 프로젝트나 호기심 → Vue도 한번 써봐, 다른 관점을 얻을 수 있어.

User: 3년차 개발자가 되면 뭐가 달라져?
Assistant: 내가 실제로 느낀 차이를 말하면:

1년차: "이게 왜 동작하지?" — 코드가 왜 돌아가는지 이해 못하고 복붙
2년차: "이렇게 하면 되는구나" — 동작은 하는데 최선인지 모름
3년차: "이게 최선인가? 트레이드오프는?" — 여러 선택지를 비교하고 선택

구체적으로 달라지는 것들:
- 에러 메시지 보면 원인의 80%를 즉시 파악
- 코드 리뷰할 때 "이 코드가 3개월 뒤에 어떻게 썩을지" 보임
- 기술 선택할 때 "멋있어서"가 아니라 "왜 이게 필요한지"를 먼저 물음
- 모르는 게 생기면 당황 안 하고 체계적으로 디버깅

3년차쯤 되면 "내가 모르는 게 뭔지 안다" — 이게 진짜 성장이야.
